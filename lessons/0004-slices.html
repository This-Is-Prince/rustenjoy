Slices

Borrowed views into collections — and the &str vs String split that confuses everyone

Primary source for this lesson The Rust Book — Ch. 4.3: The Slice Type  · This lesson follows the same motivating example as the book (first word in a string). Read after.

What is a slice?

A slice is a reference to a contiguous subsequence of a collection — part of a string, part of a vector, part of an array — without copying the data and without taking ownership. Slices are references. Everything you learned about borrowing applies to them.

There are two slice types you'll use constantly:

  • &str — a string slice (view into UTF-8 string data)
  • &[T] — a slice of any type T (view into an array or Vec)

The & is not decoration — it's telling you what a slice is: a borrow. A slice cannot outlive the collection it came from. The borrow checker enforces this.

The motivating problem — without slices

Say you want to find the first word in a string. Without slices, you might return an index:

Rust — index approach (fragile)
fn first_word_index(s: &String) -> usize {
    let bytes = s.as_bytes();
    for (i, &byte) in bytes.iter().enumerate() {
        if byte == b' ' {
            return i;   // index of first space
        }
    }
    s.len()
}

fn main() {
    let mut s = String::from("hello world");
    let word_end = first_word_index(&s);    // word_end = 5

    s.clear();          // s is now ""
    // word_end is still 5 — but it refers to nothing meaningful now
    // No compile error. Silent bug.
}

The index becomes stale the moment the string changes. The compiler can't see the connection between the index and the string — they're separate values.

Slices fix this by returning a reference tied to the original string. If the string changes while the slice is active, the borrow checker rejects it at compile time.

Rust — slice approach (borrow-checked)
fn first_word(s: &str) -> &str {
    let bytes = s.as_bytes();
    for (i, &byte) in bytes.iter().enumerate() {
        if byte == b' ' {
            return &s[..i];   // slice: 0 up to i
        }
    }
    &s[..]   // whole string
}

fn main() {
    let mut s = String::from("hello world");
    let word = first_word(&s);   // word borrows s

    s.clear();                   // ✗ COMPILE ERROR
                                 // s.clear() needs &mut s
                                 // but word holds &s — aliasing rule violation
    println!("{}", word);
}

The bug that was invisible with an index is now a compile error. The slice carries the borrow — the compiler knows word is reading from s, so it won't let you mutate s while word is alive.

Slice syntax

A slice is written as &collection[start..end] where start is inclusive and end is exclusive (same convention as Go's s[1:3]).

From start to 5&s[..5] or &s[0..5]
From 6 to end&s[6..] or &s[6..s.len()]
From 2 to 5 (exclusive)&s[2..5] → chars at 2, 3, 4
Whole string&s[..] — equivalent to &s

UTF-8 caution: string slice indices are byte offsets, not character offsets. Slicing in the middle of a multibyte character panics at runtime. For ASCII strings (most CLI input) this is a non-issue. For arbitrary Unicode, use .char_indices() to find safe boundaries.

What a slice looks like in memory

A slice is a "fat pointer" — it stores two words on the stack: a pointer to the start of the data, and a length. No heap allocation. No copy.

// String: owns heap data s (String) ptr ──→ [ h | e | l | l | o | | w | o | r | l | d ] len: 11 cap: 11 // &str slice: fat pointer into the same heap data word (&str) // = &s[..5] ptr ──→ [ h | e | l | l | o ] ↑ same memory, no copy len: 5

This is the same pattern Go uses for slice headers (ptr + len + cap), except Rust string slices don't carry a capacity because they're immutable views. The cost of passing a &str is passing two machine words — as cheap as it gets.

String vs &str — the one distinction that trips everyone up

This is the most common confusion for developers new to Rust. Here it is directly:

String literals are &str

This surprises many people. When you write "hello" in Rust source code, the type is &str — not String. The bytes are compiled into the binary itself (the data segment), and the literal is a slice pointing into that static memory.

Rust — literal type
let s: &str = "hello";    // &'static str — lives in the binary, forever
                           // no heap allocation

let s: String = String::from("hello");  // heap allocation — owned copy
let s: String = "hello".to_string();    // same thing, different syntax

The right function signature

If your function only needs to read a string, take &str, not &String. A &str accepts both a &String and a string literal — it's more flexible. A &String only accepts a &String.

Rust — prefer &str in function signatures
// Less flexible — only accepts &String
fn greet_bad(name: &String) { println!("Hello, {name}"); }

// More flexible — accepts both &String and &str literals
fn greet(name: &str) { println!("Hello, {name}"); }

fn main() {
    let owned = String::from("Alice");
    greet(&owned);         // &String coerces to &str automatically
    greet("Bob");          // string literal — already &str
}

Rule of thumb: use String when you need to own or grow a string. Use &str when you just need to read one. This maps well to Go: String ≈ []byte (mutable, owned), &str ≈ Go's string (immutable view).

Array and Vec slices — &[T]

The same concept applies to arrays and Vec. A &[T] is a borrowed view into a contiguous sequence of T values.

Rust — array and Vec slices
let a = [1, 2, 3, 4, 5];           // array, lives on the stack
let slice: &[i32] = &a[1..3];      // slice: [2, 3], no copy
println!("{:?}", slice);            // [2, 3]

let v = vec![10, 20, 30, 40];      // Vec, lives on the heap
let chunk: &[i32] = &v[..2];       // slice: [10, 20]
println!("{:?}", chunk);            // [10, 20]

The same aliasing rule applies: if you hold a &[T] slice into a Vec, you cannot mutate the Vec while the slice is alive — because the Vec might reallocate, dangling the slice. You saw this in Lesson 3 with v.push().

Functions that take slices are more generic

Rust — &[T] accepts arrays and Vecs
fn sum(values: &[i32]) -> i32 {
    values.iter().sum()
}

let arr = [1, 2, 3];
let vec = vec![4, 5, 6];

sum(&arr);     // array reference coerces to &[i32]
sum(&vec);     // Vec reference coerces to &[i32]

The pattern: just as &str is the flexible version of &String, &[T] is the flexible version of &Vec<T>. Prefer slices in function parameters for maximum flexibility.

Slices as the bridge to lifetimes

Look at the function signature from earlier:

fn first_word(s: &str) -> &str { ... }

The compiler is asking: the returned &str — how long is it valid? The answer seems obvious: as long as s is valid. And in this case, the compiler can figure that out on its own (it uses a rule called lifetime elision). But sometimes it can't. When the compiler needs help, you annotate lifetimes explicitly:

fn first_word<'a>(s: &'a str) -> &'a str { ... }

Read this as: "for some lifetime 'a, I take a &str valid for 'a and return a &str also valid for 'a." That's Lesson 5. Slices are where lifetimes first become visible in real code.

Quiz — no looking back

6 questions

1. What does a &str slice actually store on the stack?

Correct. A &str is a "fat pointer": two machine words — a pointer to the start of the UTF-8 data, and the length in bytes. No capacity, because slices are read-only views (they can't grow).
A &str is a fat pointer: pointer + length. No capacity field (that's String's job — slices can't grow, so capacity is irrelevant). No copy of the data — just a reference into existing memory.

2. What is the type of the literal "hello" in Rust?

Correct. String literals are &'static str — a slice pointing to bytes baked into the compiled binary. No heap allocation happens. You need String::from("hello") or "hello".to_string() to get an owned, heap-allocated String.
String literals have type &str (specifically &'static str — they live in the binary). They are not String — no heap allocation happens. You need String::from("hello") to get an owned String.

3. Why should function parameters prefer &str over &String?

Correct. A &String automatically coerces to &str (Deref coercion), so &str accepts both. &String only accepts an explicit reference to a String. Taking &str makes your function callable with literals, slices, and owned strings alike — more composable with no cost.
&str is more flexible: a &String coerces to &str automatically (via Deref), so any &String works as &str. But &String doesn't accept string literals or arbitrary &str slices. Prefer &str in function signatures for maximum flexibility.

4. You hold a &[i32] slice into a Vec<i32>. You try to push to the Vec. What happens?

Correct. The slice borrows the Vec immutably. push() needs a mutable borrow. Aliasing rule violation — compile error. The real danger it prevents: if push() reallocates the Vec's buffer, the slice would be a dangling pointer. The borrow checker catches the rule violation before the memory corruption.
Compile error. The &[i32] slice holds a shared borrow of the Vec. push() needs a mutable borrow — aliasing rule violation. The deeper danger: push() might reallocate, dangling the slice. Rust rejects this at compile time.

5. String is to &str as Vec<T> is to…?

Correct. Vec<T> is an owned, growable collection. &[T] is a borrowed, read-only view into any contiguous T sequence — just as &str is a borrowed view into any string data. Prefer &[T] in function parameters for the same reason you prefer &str over &String.
&[T] is the answer. Vec<T> : &[T>] follows the exact same owned-vs-borrowed relationship as String : &str. &[T] is a borrowed, read-only view into a contiguous sequence of T — it accepts &Vec<T> and &[T; N] alike.

6. Why does returning a &str from a function sometimes require a lifetime annotation?

Correct. When a function has one reference input, the compiler applies elision rules and figures out the lifetime. When there are multiple input references and the output is a reference, the compiler can't tell which input the output came from — so you annotate it explicitly. That's what Lesson 5 is about.
With one input reference, the compiler infers the lifetime via elision rules. With multiple input references and a reference return, the compiler can't determine which input the returned reference came from — so you must annotate. Lifetime annotations are the topic of Lesson 5.

What's next

You've seen slices introduce a natural question: how long is this reference valid? The compiler answers it silently in simple cases (one input reference → output reference) but needs your help when the situation is ambiguous. Lesson 5 names that concept — lifetimes — and shows you the annotation syntax that makes the compiler's implicit reasoning explicit. This is the last piece of the ownership puzzle.
