Lifetimes

Naming what the compiler already knows — and the one thing it sometimes can't figure out alone

Primary source for this lesson The Rust Book — Ch. 10.3: Validating References with Lifetimes  · This is a longer chapter. Read it after — the lesson gives you the mental model first.

The one thing lifetimes are not

Before anything else: lifetime annotations do not control how long values live. The compiler already knows exactly how long every value lives — that's what ownership and scopes give it. Lifetimes are not a mechanism for extending or shortening that.

Lifetime annotations do exactly one thing: they describe relationships between references. Specifically: "this output reference is valid for at least as long as this input reference." They give the compiler vocabulary to describe constraints it already needs to enforce.

The right mental model: a lifetime annotation is like a label you stick on two references that says "these two must live at least as long as each other." You're not changing anything about how the program runs. You're helping the compiler prove a property it already wants to verify.

Why they're needed — the problem

The compiler can verify that references are valid on its own for simple cases. But consider this:

Rust — compile error: ambiguous lifetime
fn longest(x: &str, y: &str) -> &str {
    if x.len() > y.len() { x } else { y }
}
// error[E0106]: missing lifetime specifier
// the compiler doesn't know if the return is tied to x or y

The compiler sees: you're returning a reference, and you have two input references. It cannot tell — from the function signature alone — whether the returned reference came from x or from y. It needs to know, because the caller needs to know how long the returned reference is valid for. If it came from x, the caller must keep x alive. If from y, keep y alive. If from either, keep both alive.

The function body would tell the compiler which — but Rust checks call sites using only the signature, not the body. (This is what makes separate compilation possible.) So you must encode the relationship in the signature itself.

Lifetime annotation syntax

A lifetime parameter is a name starting with ' (apostrophe), placed in angle brackets after the function name, and used on the references you want to connect.

fn longest<'a>(x: &'a str, y: &'a str) -> &'a str
↑ declare
the name ↑ x borrows
for at least 'a ↑ y borrows
for at least 'a ↑ return lives
for at most 'a

Read this as: "for some lifetime 'a — whatever the shorter of x's and y's lifetimes turns out to be — the return value is valid for that same duration." You're not choosing a duration. You're saying: whatever duration makes both inputs valid, the output lasts exactly that long.

Rust — annotated, compiles
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() { x } else { y }
}

fn main() {
    let s1 = String::from("long string");
    let result;
    {
        let s2 = String::from("xy");
        result = longest(s1.as_str(), s2.as_str());
        println!("{}", result);   // ✓ fine — result used inside s2's scope
    }
    // println!("{}", result);    // ✗ would error — s2 dropped, result might point to it
}

The compiler now knows: result is valid for 'a, which is the shorter of s1's and s2's lifetimes. When s2 is dropped, 'a ends, and using result after that is a compile error. No dangling pointer.

The lifetime parameter 'a is not a duration you pick. It's a variable — the compiler substitutes in the actual lifetimes at each call site and checks that the constraints hold. If they don't, it's a compile error at the call site. The function author isn't saying "this lasts 5 seconds"; they're saying "the output lasts as long as both inputs do."

Lifetime elision — why most code has no annotations

If annotations were required everywhere, Rust would be unbearable. Early Rust did require them everywhere. The team noticed that the same patterns appeared over and over, and encoded them as elision rules — patterns the compiler recognises and fills in automatically.

The Three Elision Rules (applied in order)

  1. Each reference parameter gets its own distinct lifetime. fn foo(x: &str, y: &str) → compiler treats as fn foo<'a, 'b>(x: &'a str, y: &'b str)
  2. If there is exactly one input lifetime, it is assigned to all output references. fn first(s: &str) -> &str → compiler treats as fn first<'a>(s: &'a str) -> &'a str
  3. If one of the inputs is &self or &mut self, its lifetime is assigned to all outputs. Methods on structs almost never need lifetime annotations on their return values.

If these three rules fully determine all lifetimes in the signature, the compiler fills them in and you write nothing. If they leave any output reference's lifetime ambiguous, you must annotate. That's the only situation you need annotations on functions — and it's rarer than tutorials make it seem.

Rust — elision handles these automatically
// Rule 2: one input ref → output gets same lifetime. No annotation needed.
fn first_word(s: &str) -> &str { ... }

// Rule 3: &self method → output gets self's lifetime. No annotation needed.
impl ImportantExcerpt {
    fn level(&self) -> i32 { ... }
    fn announce(&self, announcement: &str) -> &str { self.part }
}

Lifetimes in structs

If a struct holds a reference, it needs a lifetime annotation — because the struct cannot outlive the data it's borrowing.

Rust — struct with a reference
struct Excerpt<'a> {
    part: &'a str,       // this struct borrows a string slice
}

fn main() {
    let novel = String::from("Call me Ishmael. Some years ago...");
    let first_sentence = novel.split('.').next().unwrap();

    let excerpt = Excerpt { part: first_sentence };
    // excerpt borrows from novel — excerpt cannot outlive novel
    println!("{}", excerpt.part);
}
// novel dropped here — excerpt was dropped first (declared after), so fine

The 'a on the struct says: "an instance of Excerpt cannot outlive the string slice its part field references." The compiler enforces this at every use site.

Practical advice: structs holding references are more complex to work with. In most application code, prefer owning your data (store a String, not a &str) to avoid needing lifetime annotations on structs. Use reference-holding structs when performance is critical and copying is too expensive. You'll know when you need it.

The 'static lifetime

'static is a special lifetime meaning "valid for the entire program." You've already been using 'static without knowing it — string literals have type &'static str because their bytes are embedded in the binary and live forever.

Rust — 'static
let s: &'static str = "I live for the whole program";
// This is the same as:
let s = "I live for the whole program";   // type inferred as &'static str

When the compiler says "try adding 'static": this is usually a hint that you've structured something incorrectly, not an invitation to slap 'static on everything. If you need 'static on a function argument and you're not dealing with thread spawning or trait objects, step back — there's likely a design issue.

Go and C don't have this — why?

GoCRust
Reference validity tracked? By the GC at runtime Not at all — your problem By the borrow checker at compile time
Dangling reference possible? No — GC keeps data alive Yes — classic bug No — compile error
Lifetime annotations? None needed (GC handles it) None (no checking) Sometimes needed for ambiguous cases
Cost of the mechanism GC runtime overhead None (no safety) Compile-time only — zero runtime cost

Go doesn't need lifetimes because the GC keeps referenced data alive automatically — a pointer is valid as long as anything points to it, and the GC scans to find out. C doesn't need lifetime syntax because it provides no safety. Rust provides safety and no GC, so it needs the compile-time proof — and lifetime annotations are the syntax for writing that proof.

Pulling the ownership arc together

You've now covered the full ownership system across five lessons. Here's how the pieces connect:

ConceptWhat it doesWhat it prevents
OwnershipOne owner per value; freed when owner dropsDouble-free, memory leaks
Move semanticsAssignment transfers ownershipDouble-free from aliased owners
BorrowingTemporary read/write access via referencesUse-after-free, data races, iterator invalidation
SlicesBorrowed views into contiguous dataStale index bugs, dangling string pointers
LifetimesAnnotate relationships between referencesDangling references across function boundaries

Every rule traces back to Lesson 1's single invariant: every value has exactly one owner at a time, freed when the owner dies. Lifetimes are the final piece that makes this verifiable across function boundaries, where the compiler can't see the full picture from a single scope.

Quiz — no looking back

6 questions

1. What do lifetime annotations actually do?

Correct. Lifetime annotations don't change program behaviour or how long values live — the compiler already knows that from scopes. They describe constraints between references: "this output reference is valid for at most as long as these input references." The compiler uses them to verify at call sites that no dangling reference can result.
Lifetime annotations don't change how long values live — that's determined by ownership and scopes. They describe relationships: "this output reference lives at most as long as these inputs." The compiler uses that to verify, at call sites, that no dangling reference can result.

2. Why does fn longest(x: &str, y: &str) -> &str fail to compile without lifetime annotations?

Correct. With two input references and one output reference, elision rule 2 doesn't apply (it only fires when there's exactly one input reference). The compiler checks call sites using only the signature — it can't look at the body. Without knowing which input the output borrows from, it can't verify the caller keeps the right thing alive.
Elision rule 2 requires exactly one input reference. With two, the compiler has no way to know if the return came from x or y. It needs to know to verify call sites — and it only uses the function signature, not the body, for that. Hence the annotation is required.

3. In fn longest<'a>(x: &'a str, y: &'a str) -> &'a str, what does 'a actually represent at a call site?

Correct. 'a is a type variable the compiler substitutes at each call site. It resolves to the shorter of x's and y's actual lifetimes — because the return must be valid under both, and the constraint is that all three share the same 'a. The compiler checks the caller doesn't use the return after the shorter-lived input is dropped.
'a is a variable, not a fixed duration. At each call site, the compiler substitutes 'a with the shorter of x's and y's actual lifetimes — the most conservative bound that keeps the constraint "all three are 'a" satisfiable. The return is then only valid within that duration.

4. Which elision rule handles fn first_word(s: &str) -> &str automatically?

Correct. Exactly one input reference (s: &str) → Rule 2 applies: the compiler treats the return as having the same lifetime as s. Equivalent to writing fn first_word<'a>(s: &'a str) -> &'a str. No annotation needed.
Rule 2: when there's exactly one input reference, the output reference is assigned that same lifetime automatically. first_word has one input (&str) and one output (&str) — the compiler fills in the 'a without you writing it.

5. A struct holds a &str field. Why does it need a lifetime parameter?

Correct. If a struct holds a reference, the compiler needs to know the relationship: the struct cannot outlive the data it references. The lifetime parameter on the struct encodes this constraint, so the compiler can verify it at every use site.
A struct holding a reference creates a borrowing relationship: the struct must not outlive the data it borrows. The lifetime annotation on the struct tells the compiler about this constraint so it can be verified at every use site. It's required — not optional.

6. When the compiler suggests adding 'static to fix a lifetime error, what should you do first?

Correct. 'static means "valid for the whole program." If you're reaching for it on a function argument or struct field (outside thread spawning or trait objects), it almost always means the design needs rethinking — perhaps the struct should own its data (String instead of &str) rather than borrow it.
The compiler's 'static suggestion is a mechanical fix, not necessarily the right design choice. 'static means "lives for the entire program." If you're not dealing with threads or trait objects, needing 'static usually signals a design issue — often the fix is to own the data (String) rather than borrow it (&str).
