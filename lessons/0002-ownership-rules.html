Ownership Rules

What "owner" means in code, move semantics, and the Copy/Clone divide

Primary source for this lesson The Rust Book — Ch. 4.1: What Is Ownership?  · Read (or re-read) after this lesson — the book covers this same ground with runnable playground links.

The three rules

Last lesson established the why. Now the what. Rust's entire ownership system is built on exactly three rules. Everything else — borrowing, lifetimes, smart pointers — is just these rules playing out in more complex situations.

Rust Ownership Rules

  1. Each value in Rust has an owner. Every piece of data is "owned" by exactly one variable, struct field, or data structure.
  2. There can only be one owner at a time. Ownership is exclusive — you can't have two owners. This is what prevents double-free.
  3. When the owner goes out of scope, the value is dropped. "Dropped" = freed. The compiler inserts the free() at the closing brace of the owner's scope.

These three sentences are the entire mental model. Let's watch them play out in code.

Scope — the simplest case

A value is created when its owner is declared. It's dropped when the owner's scope ends.

Rust
{
    let s = String::from("hello");   // s owns the String
    //   ↑ allocated here, s is the owner

    // ... use s ...

}   // ← s goes out of scope here
    //   Rust calls drop(s) — the memory is freed
    //   This is where the compiler inserts the free()

If you know C, this is exactly a free() at the closing brace, inserted automatically. If you know Go, the memory would stay alive until the next GC cycle. In Rust, it's deterministic: freed at the exact point the scope closes, no runtime involvement.

Move semantics — the part that surprises Go developers

In Go, assigning a value copies it (for value types) or copies the pointer (for reference types). Either way, both variables are usable after the assignment. In Rust, assigning a heap-allocated value moves ownership. The original variable is invalidated.

Go — this works fine
s1 := "hello"
s2 := s1        // copy of the string header; both are valid
fmt.Println(s1) // ✓ works
fmt.Println(s2) // ✓ works
Rust — this is a compile error
let s1 = String::from("hello");
let s2 = s1;          // ownership MOVES from s1 to s2
                      // s1 is now invalidated
println!("{}", s1);   // ✗ ERROR: value borrowed after move
println!("{}", s2);   // ✓ fine — s2 is the owner now

This is the thing that shocks people coming from Go. Why? Because Rust is protecting Rule 2: only one owner at a time. If both s1 and s2 owned the same heap data, they'd both try to free it when they go out of scope — a double-free. So Rust makes ownership exclusive by invalidating the source.

let s1 = String::from("hello"); s1 ──→ [ heap: "hello" ] // s1 owns it let s2 = s1; // MOVE s1 [ heap: "hello" ] // s1: invalidated s2 ──────────────────→ ↑ // s2 owns it now

The heap data doesn't move — the ownership does. Rust didn't copy the string; it transferred the title deed.

The C comparison: in C, you could copy a pointer and have two pointers to the same memory. Rust's move makes that illegal for owned values — because if both were owners, both would free the memory. Move semantics are Rust saying "I'll only let one variable hold the title."

Moves through functions

Passing a value into a function is a move. Returning a value from a function is also a move. This surprises most people the first time.

Rust
fn greet(name: String) {           // name takes ownership of the String
    println!("Hello, {}!", name);
}   // name goes out of scope here — String is dropped

fn main() {
    let s = String::from("Alice");
    greet(s);                      // s is MOVED into greet
    println!("{}", s);             // ✗ ERROR: s was moved
}

Coming from Go this feels brutal — in Go you just pass the string, use it, and it's fine. But remember what Rust is preventing: two things trying to free the same memory. The function took ownership, so it will free the String when it returns. The caller can no longer touch it.

If you want to keep using s after the call, you have two options. The first (which you shouldn't reach for by default) is to return ownership back:

Rust — return ownership back (awkward, but valid)
fn greet(name: String) -> String {  // takes and returns ownership
    println!("Hello, {}!", name);
    name                            // moves ownership back to caller
}

fn main() {
    let s = String::from("Alice");
    let s = greet(s);              // s is moved in, then moved back out
    println!("{}", s);             // ✓ fine now
}

This is the "right problem, wrong solution." The real solution is borrowing — which is Lesson 3. For now, just understand that functions participate in the ownership system exactly like variable assignments do.

Copy types — the exception to the rule

If every assignment moved ownership, using integers would be unbearable. Rust has an escape hatch: the Copy trait.

Types that implement Copy are duplicated on assignment — no move, no invalidation — because they live entirely on the stack and are cheap to copy.

Copy types (assignment copies)Non-Copy types (assignment moves)
i32, u64, f64, all numeric typesString
boolVec<T>
charBox<T>
Tuples/arrays of Copy types: (i32, bool)Any type that owns heap memory
&T (shared references)Most custom structs (unless you derive Copy)
Rust — Copy type: no move
let x = 5;        // i32 — Copy type
let y = x;        // x is COPIED, not moved
println!("{}", x); // ✓ x is still valid
println!("{}", y); // ✓ y has an independent copy
Rust — non-Copy type: moves
let s1 = String::from("hi");  // String — non-Copy (owns heap data)
let s2 = s1;                  // s1 is MOVED, not copied
// println!("{}", s1);        // ✗ would error — s1 moved

The heuristic: if a type owns data on the heap, it's non-Copy. If it lives entirely on the stack with a known, fixed size, it's probably Copy. The reason: copying a stack integer is trivially cheap (just copy the bytes). Copying a String means allocating new heap memory and copying the contents — a non-trivial operation that Rust won't do silently.

Explicit deep copy: clone()

If you actually need two independent owned copies of a heap value, ask for it explicitly with .clone(). Rust won't do it silently because it's not cheap.

Rust
let s1 = String::from("hello");
let s2 = s1.clone();          // explicitly deep copies the heap data

println!("{}", s1);           // ✓ s1 still valid — it wasn't moved
println!("{}", s2);           // ✓ s2 has its own independent copy

Go comparison: In Go, strings.Clone(s) exists but you rarely need it because strings are immutable and the runtime handles sharing. In Rust, clone() is visible and deliberate — it's a signal in code that says "yes, I know I'm allocating here." This aligns with Rust's philosophy: no hidden allocations.

The drop function

When an owner goes out of scope, Rust calls its drop() method automatically. You can call it manually to free early — useful for releasing a file handle, lock, or connection before the scope ends.

Rust
let s = String::from("hello");
drop(s);                  // freed early — s is now gone
// println!("{}", s);     // ✗ would error: s was moved into drop()

You'll rarely call drop() manually. It's here so you know: "drop" is the word Rust uses for "the owner left, free the memory."

Putting it together — reading a compile error

When the borrow checker rejects your code, it's almost always enforcing one of these three rules. Here's how to read the error:

Rust compiler error
error[E0382]: borrow of moved value: `s1`
  --> src/main.rs:4:20
   |
2  |     let s1 = String::from("hello");
   |         -- move occurs because `s1` has type `String`
3  |     let s2 = s1;
   |              -- value moved here
4  |     println!("{}", s1);
   |                    ^^ value borrowed here after move

Translate this: "you moved s1 on line 3 (ownership transferred to s2), so on line 4, s1 no longer owns anything. I won't let you use it." The fix is either: don't use s1 after the move, use clone() on line 2, or — lesson 3 — use a reference instead of moving at all.

Quiz — no looking back

6 questions

1. After let s2 = s1; where s1 is a String, what happens to s1?

Correct. Assigning a non-Copy type moves ownership. s1 is invalidated — the compiler will reject any subsequent use of it. s2 is now the sole owner.
Not quite. Assignment of a non-Copy type is a move, not a copy or a share. s1 is invalidated after the move — neither empty nor co-owner. Use .clone() if you need two independent copies.

2. Which of these types uses Copy semantics on assignment?

Correct. i32 is a fixed-size stack value — trivially cheap to copy. All numeric primitives, bool, and char are Copy. Types that own heap memory (String, Vec, Box) are non-Copy.
String, Vec, and Box all own heap memory — they use move semantics. i32 is a fixed-size stack integer, trivially copyable, and implements the Copy trait.

3. What does s.clone() do differently from a plain assignment?

Correct. clone() allocates new heap memory and copies the contents into it — s keeps its own copy, the result is an independent new copy. This is why Rust requires you to call it explicitly: it's not free.
clone() performs a deep copy: it allocates new heap memory and copies the data. s remains the owner of the original. The result is a fully independent copy. Plain assignment moves (transfers the title without copying data).

4. You pass a String to a function. After the call returns, can you use the original variable?

Correct. Passing to a function is a move. The function owns the String, and when the function's scope ends, it's dropped. The caller can no longer use the original. The proper fix is borrowing (Lesson 3) — passing a reference instead of ownership.
Passing a String to a function moves ownership into the function. The function owns it, and when the function returns (and its scope closes), the String is dropped. The caller's variable is invalidated. The right fix is a reference — covered in Lesson 3.

5. Rule 2 says only one owner at a time. What specific memory bug does this directly prevent?

Correct. If two owners existed for the same value, both would try to free it when their scopes ended — a double-free. Single ownership means single free, always.
Single ownership prevents double-free: if only one variable can own the data, only one scope can trigger the free. Stack overflows, null pointers, and integer overflow are separate concerns.

6. When exactly does Rust free (drop) a value?

Correct. Drop timing is static and deterministic: the compiler knows the owner's scope at compile time, and inserts the free at the closing brace. No GC, no explicit call, no runtime tracking — pure compile-time analysis.
Rust has no GC and no explicit free(). Drop timing is determined at compile time: the compiler inserts the free at the closing brace of the owner's scope. This is what makes Rust memory management both safe and zero-cost.

What's next

You now understand ownership and move semantics — including why functions are so greedy. But the problem is obvious: if every function call consumed your data, writing programs would be a nightmare of clones and return-ownership chains.

The solution is borrowing: giving a function temporary access to a value without transferring ownership. That's Lesson 3 — and it's where the borrow checker's real personality comes out.
