The Memory Problem

Why does ownership exist? What was broken before it?

Primary source for this lesson The Rust Book — Ch. 4.1: What Is Ownership?  · Read this after the lesson to reinforce with the official framing.

The setup

Every program needs memory — to store variables, build data structures, pass values between functions. The question every language designer faces is: who is responsible for that memory, and when does it get freed?

There are exactly three answers that have seen widespread adoption. You've lived with two of them already.

Answer 1: You manage it (C)

In C, you call malloc() to ask the OS for memory. When you're done, you call free(). The contract is simple. The execution is brutal.

C
char *name = malloc(64);   // ask for 64 bytes
strcpy(name, "Alice");
// ... use name ...
free(name);                // give it back

// What could go wrong?
printf("%s\n", name);      // USE AFTER FREE — undefined behavior
free(name);                // DOUBLE FREE — often crashes
// (or just forget free() entirely — MEMORY LEAK)

These aren't edge cases. They're the three failure modes that have caused the majority of security vulnerabilities in C programs for 50 years: use-after-free, double-free, and memory leaks. They're hard to catch because the compiler doesn't check them — only the runtime (or a fuzzer, or an attacker) will tell you something went wrong.

The C promise: you have total control, total performance, and total responsibility. The compiler trusts you completely. That trust is frequently misplaced.

Answer 2: A runtime manages it (Go)

Go takes the opposite philosophy: the programmer shouldn't have to think about memory at all. A garbage collector (GC) runs in the background, tracks which values are still reachable, and frees anything that isn't.

Go
name := "Alice"   // allocated — runtime tracks it
// ... use name ...
// no free() needed — GC will clean up when name goes out of scope

This is much safer. You cannot have a use-after-free or double-free in Go. The GC simply won't free memory until nothing points to it.

But the GC doesn't come free. It has costs:

  • Runtime overhead: the GC is a background program running alongside yours
  • Pause latency: when the GC runs a collection cycle, your program stops (even briefly)
  • Unpredictability: you can't control when memory is freed
  • Memory overhead: the GC needs headroom to work efficiently — programs often use more RAM than their live data requires

For most Go programs — web servers, CLIs, tooling — these tradeoffs are completely acceptable. But they make Go unsuitable for certain contexts: real-time systems where a GC pause is catastrophic, embedded systems where you have 32KB of RAM, or kernels where there's no runtime to host the GC.

The Go promise: memory safety by default, at the cost of a runtime and non-deterministic collection. You trade control for safety.

Answer 3: The compiler manages it (Rust)

Rust's designers asked: what if we could get C's performance and control, and Go's safety — but without a GC?

The answer they found is ownership. Instead of tracking memory at runtime (like Go's GC does), Rust tracks memory at compile time through a set of rules. The compiler — not a background thread — knows exactly when every piece of memory will be freed, and inserts the free() calls automatically.

No GC. No runtime. No pauses. No unsafe manual free. Just rules.

Rust
let name = String::from("Alice");   // allocated
// ... use name ...
// name goes out of scope here — Rust inserts the free() at compile time
// No GC. No malloc/free. Impossible to double-free or use-after-free.

The catch? The rules are enforced by a compiler component called the borrow checker. The borrow checker rejects programs that could cause memory errors — at compile time, before your code ever runs. This is why fighting the borrow checker feels so strange: it's not a test suite catching bugs at runtime, it's a proof system catching them before compilation.

The Rust promise: memory safety and C-level performance, enforced at compile time by the ownership system. You trade compile-time complexity for freedom from both a GC and manual memory bugs.

The one rule that makes it work

Rust's whole ownership system flows from one fundamental rule:

Every value has exactly one owner at a time.

When the owner goes out of scope, the value is freed. Automatically. Unconditionally.

That's it. That single invariant is enough for the compiler to know exactly when to insert the free(). No double-frees (only one owner can free it). No use-after-free (the value is freed when the owner dies — you can't use a dead owner). No GC needed (the compiler sees all the scopes statically).

The next lessons will unpack what "owner" means, what happens when you move or copy a value, and how borrowing lets you temporarily share access without transferring ownership. But every one of those rules exists to protect this single invariant.

Why the borrow checker feels alien: In Go, the compiler lets you do almost anything with values — the GC sorts it out. In C, the compiler doesn't check memory at all. Rust is the first language where the compiler has a strong opinion about who is allowed to access a value and when. It's not being difficult. It's enforcing a proof.

Quiz — check your understanding

5 questions · no looking back

1. In C, calling free(ptr) and then using ptr afterward is called what?

Correct. Use-after-free: you freed the memory, then tried to read it. The pointer is now "dangling" — it points to freed memory that could be reused for anything.
Not quite. Using freed memory is called use-after-free. A double-free is calling free() twice on the same pointer. A memory leak is forgetting to free at all.

2. Go's garbage collector prevents use-after-free errors. What does it cost?

Correct. The GC is a runtime program that adds overhead and periodically pauses your code to scan for unreachable memory. Safe, but not free.
That's Rust's cost (compile-time complexity), not Go's. Go's tradeoff is runtime overhead: a background GC that adds memory and CPU cost, and occasional pause latency.

3. Rust inserts free() calls with no runtime GC. How does it know when to free?

Correct. Ownership is a compile-time concept. The borrow checker proves at compile time when each value's owner goes out of scope, and the compiler inserts the free there — no runtime tracking needed.
Reference counting (Rc/Arc) is one Rust pattern, but it's not how basic ownership works. The core mechanism is compile-time: the compiler statically knows when each owner leaves scope and inserts the free there.

4. "Every value has exactly one owner at a time." What memory error does this rule directly prevent?

Correct. If only one entity owns a value, only that entity can free it — making double-free structurally impossible. Use-after-free is prevented by a different rule (the value is freed when the owner dies, and the compiler won't let you use a dead owner).
The single-owner rule most directly prevents double-free: if only one thing owns the memory, it can only be freed once. Stack overflow and null pointers are separate concerns.

5. Why does the Rust borrow checker feel unlike anything in Go or C?

Correct. In C the compiler doesn't check memory access at all; in Go the GC handles it at runtime. Rust is the first mainstream language where the compiler itself has strong, statically-enforced rules about value access — which is exactly why it feels foreign at first.
The key distinction is that the borrow checker is a compile-time proof system about value access — something neither C nor Go's compiler does. C doesn't check memory; Go delegates it to a runtime GC.
