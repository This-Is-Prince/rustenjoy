Borrowing

Temporary access without transferring ownership — and the aliasing rule that makes it safe

Primary source for this lesson The Rust Book — Ch. 4.2: References and Borrowing  · Read after the lesson. The book has runnable Playground examples for every rule here.

The problem borrowing solves

End of Lesson 2, you were stuck: passing a String to a function moved it, and you couldn't use it afterward. The only fix was to clone (allocate) or return ownership back (awkward). Neither is acceptable as the default.

What you actually want is to lend the function access — let it read or write the value — without giving up ownership. That's borrowing. A borrowed value is accessed through a reference, written with &.

Rust — borrow instead of move
fn print_len(s: &String) {          // takes a reference, not ownership
    println!("length: {}", s.len());
}   // s (the reference) goes out of scope here — the String is NOT dropped
    // because this function never owned it

fn main() {
    let owned = String::from("hello");
    print_len(&owned);              // lend a reference
    println!("{}", owned);          // ✓ still valid — we never moved it
}

The & in &String says "a reference to a String" — not the String itself. The function borrows access; ownership stays with owned in main. When the function returns, the borrow ends. The String is not dropped.

Mental model: a reference is like handing someone your house key. They can get in, but they don't own the house. When they give the key back (the borrow ends), you still own the house. Rust tracks who holds your key at compile time.

Two kinds of reference

Rust has exactly two kinds of reference, and they have fundamentally different rules.

Shared reference &T Mutable reference &mut T
Syntax &value / &T &mut value / &mut T
Can read? Yes Yes
Can write? No Yes
How many at once? Unlimited Exactly one
Can coexist with &mut? No No
Rust — shared reference (read-only)
let s = String::from("hello");

let r1 = &s;    // shared borrow
let r2 = &s;    // another shared borrow — fine, unlimited readers

println!("{} and {}", r1, r2);  // ✓ both valid simultaneously
Rust — mutable reference (read+write)
let mut s = String::from("hello");   // note: variable must be `mut`

let r = &mut s;                      // mutable borrow
r.push_str(", world");               // ✓ can modify through the reference
println!("{}", r);                   // ✓ prints "hello, world"

The aliasing rule — the heart of the borrow checker

The single most important rule in Rust's reference system:

The Aliasing Rule

  • Many shared references, OR one mutable reference. At any given point in time, you can have either multiple &T borrows, or exactly one &mut T borrow — but never both simultaneously.
✓ Legal — many readers
let r1 = &s;
let r2 = &s;
// both active — fine
✗ Illegal — reader + writer
let r1 = &s;
let r2 = &mut s;
// compile error
Rust — borrow checker rejects this
let mut s = String::from("hello");

let r1 = &s;          // shared borrow starts
let r2 = &mut s;      // ✗ ERROR: cannot borrow `s` as mutable
                      //   because it is also borrowed as immutable
println!("{}", r1);

Why? What's the actual danger? Two things:

Danger 1: Data races

A data race happens when two things access the same memory concurrently and at least one writes. In Go, the race detector catches this at runtime. In Rust, the aliasing rule prevents it at compile time — you literally cannot express a data race in safe Rust, because you can never have a writer active at the same time as any reader.

Go — race detector catches this at runtime
// Go has no compile-time protection; needs -race flag at runtime
go func() { s = "modified" }()   // writer
go func() { fmt.Println(s) }()   // reader — race!
Rust — prevented at compile time
// In safe Rust, this class of bug cannot be expressed:
// You cannot have &mut s and &s active simultaneously — ever.
// The borrow checker enforces this even in single-threaded code.

Danger 2: Iterator invalidation

This is the bug that C++ developers know intimately. If you're iterating over a vector and something modifies that vector (adds or removes elements, causing a reallocation), your iterator is now pointing at freed or moved memory.

C — undefined behavior
// Classic C++ invalidation (analogous):
for (auto it = vec.begin(); it != vec.end(); ++it) {
    vec.push_back(42);  // reallocates — it is now a dangling pointer
    // reading *it is undefined behavior
}
Rust — compile error, not UB
let mut v = vec![1, 2, 3];
let first = &v[0];       // shared borrow of v
v.push(4);               // ✗ ERROR: cannot borrow `v` as mutable
                         //   because it is also borrowed as immutable
println!("{}", first);   // first could be dangling after the push

The aliasing rule made this a compile error instead of a segfault. Rust looked at the code and said: you hold a reader (first borrows v) and you're trying to write (v.push()). That violates the rule. Rejected.

The payoff: every bug shown above — data races, iterator invalidation — is caught by the borrow checker before your code compiles. You get C-level performance with none of these failure modes. This is what "fearless concurrency" means in Rust's marketing.

Borrow scopes — Non-Lexical Lifetimes (NLL)

A borrow ends when the reference is last used, not at the closing brace of its declaration scope. This is called Non-Lexical Lifetimes (NLL), and it means borrows are often shorter than you'd expect. Modern Rust (2018 edition onward) handles this for you.

Rust — borrows end at last use, not at closing brace
let mut s = String::from("hello");

let r1 = &s;
let r2 = &s;
println!("{} and {}", r1, r2);
// r1 and r2 are last used on the line above
// their borrows END here, before the mutable borrow below

let r3 = &mut s;           // ✓ fine — r1 and r2 are no longer active
r3.push_str(", world");
println!("{}", r3);

You don't need to think about this too hard — the compiler manages it. But it explains why some code that looks like it should fail actually compiles fine.

Dangling references — and why Rust prevents them

In C, returning a pointer to stack-allocated memory is a classic mistake. The memory is freed when the function returns, but the caller holds a pointer to it.

C — dangling pointer (undefined behavior)
char* get_name() {
    char name[10] = "Alice";    // stack-allocated
    return name;                // name freed on return — caller gets garbage
}

char* p = get_name();
printf("%s\n", p);             // reading freed stack memory — UB
Rust — compile error instead
fn dangle() -> &String {        // ✗ ERROR: missing lifetime specifier
    let s = String::from("hi"); // s is created here
    &s                          // try to return a reference to s
}   // s is dropped here — the reference would dangle
    // Rust rejects this at compile time

The compiler sees that the reference would outlive the value it points to, and refuses to compile. The fix — return the owned value, not a reference to it:

Rust — return the owned value
fn no_dangle() -> String {
    let s = String::from("hi");
    s                           // move ownership to the caller — valid
}

The dangling reference check is also what motivates lifetimes — the annotation system for telling the compiler how long references are valid. That's Lesson 5. For now, just know: Rust can see when a reference would outlive its value, and it rejects the code.

The Go comparison — runtime vs compile time

Go doesn't have references in Rust's sense. Go has pointers (*T) and interfaces, but the language doesn't track whether you have simultaneous readers and writers. Protection is opt-in at runtime (mutexes, channels, the race detector). Rust's protection is mandatory at compile time.

GoRust
Shared read access Any pointer can read; no enforcement &T — many allowed simultaneously
Exclusive write access sync.Mutex at runtime &mut T — compiler enforces exclusivity
Data race detection go test -race at runtime Prevented at compile time in safe Rust
Iterator invalidation Possible (not a language guarantee) Compile error
Dangling pointer Not possible (GC keeps value alive) Compile error

Quiz — no looking back

6 questions

1. What does &s create in Rust?

Correct. &s is a shared reference — it borrows read-only access to s without taking ownership. When the reference goes out of scope, the value is not dropped, because this code never owned it.
&s is a shared reference: read-only access to s, without taking ownership. No allocation, no copy, no mutation — just a borrow. The original owner retains ownership throughout.

2. You have an active &s (shared reference). Can you create a &mut s at the same time?

Correct. The aliasing rule: either many shared references, or exactly one mutable reference — never both at the same time. This is the compile-time guarantee that prevents data races and iterator invalidation.
No. The aliasing rule forbids it: you cannot have a mutable reference while any shared reference is active. The borrow checker enforces this at compile time regardless of threading.

3. How many &T (shared) references to the same value can be active simultaneously?

Correct. Shared references are unlimited — multiple readers can coexist freely. Only mutable references are restricted to one-at-a-time, and they cannot coexist with shared references.
Shared references are unlimited. The "exactly one" restriction applies to mutable references (&mut T), not shared ones. Many readers simultaneously is always safe — only writers need exclusivity.

4. In the code: let first = &v[0]; v.push(4); — why does Rust reject this?

Correct. first borrows v immutably (&v[0] borrows v). push() would need a mutable borrow of v — aliasing rule violation. The deeper danger: push() might reallocate the vector's buffer, making first a dangling pointer. Rust catches the rule violation before the memory corruption can happen.
first holds a shared borrow of v (reading v[0] borrows v). push() needs to mutably borrow v to modify it. Shared + mutable simultaneously violates the aliasing rule. The real danger is reallocation: push() might move the buffer, leaving first dangling. Rust catches the rule, preventing the corruption.

5. How does Rust prevent returning a dangling reference from a function?

Correct. The borrow checker performs a static analysis: if a reference would outlive the value it references (e.g., returning &s where s is a local variable), it's a compile error. This is enforced via lifetime analysis — the topic of Lesson 5.
No GC, no auto-copy, and references can be returned (with lifetimes). The borrow checker statically determines whether a reference could outlive its value — if it can, the code is rejected. You can return a reference to data the caller owns; you cannot return a reference to data the function owns (that gets dropped on return).

6. In Go you'd protect shared mutable state with sync.Mutex. What does Rust use instead?

Correct — for single-threaded code. The borrow checker's aliasing rules make shared mutation structurally impossible in safe Rust: you can't have &mut T and &T at the same time, so no runtime lock is needed. For multi-threaded shared state, Rust does have Mutex<T> — but the borrow checker still verifies you acquire the lock before accessing the data.
In single-threaded code, the borrow checker's aliasing rules eliminate the need for a mutex entirely: you physically cannot have a writer and reader simultaneously. For multi-threaded shared state, Rust does have Mutex<T> — but unlike Go, the borrow checker enforces that you hold the lock before accessing the protected data.

What's next

You can now borrow values without consuming them, and you understand why the aliasing rule exists (data races, iterator invalidation, dangling pointers). The borrow checker is no longer an obstacle — it's a proof system you're starting to read.
