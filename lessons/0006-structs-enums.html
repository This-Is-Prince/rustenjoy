Structs & Enums

Modelling data — and the enum that makes null impossible

Primary sources for this lesson The Rust Book — Ch. 5: Structs  &  Ch. 6: Enums and Pattern Matching

Part 1 — Structs

Rust structs look like Go structs. Named fields, grouped into a type. The main surface difference is that methods live in a separate impl block rather than being declared alongside the fields.

Go
type Rectangle struct {
    Width  float64
    Height float64
}

func (r Rectangle) Area() float64 {
    return r.Width * r.Height
}
Rust
struct Rectangle {
    width:  f64,
    height: f64,
}

impl Rectangle {
    fn area(&self) -> f64 {           // &self = shared borrow of the instance
        self.width * self.height
    }

    fn scale(&mut self, factor: f64) { // &mut self = mutable borrow
        self.width  *= factor;
        self.height *= factor;
    }

    fn new(width: f64, height: f64) -> Rectangle {  // associated function (no self)
        Rectangle { width, height }                  // field shorthand like JS
    }
}

fn main() {
    let mut r = Rectangle::new(10.0, 5.0);  // :: for associated functions
    println!("{}", r.area());               // 50
    r.scale(2.0);
    println!("{}", r.area());               // 200
}

The three self flavours

ParameterMeaningWhen to use
&selfShared borrow — read onlyMost methods; reading fields, computing values
&mut selfMutable borrow — read + writeMethods that modify the struct's fields
selfTakes ownership — consumes the valueBuilder patterns, conversions; rare in normal methods

This is the ownership system you already know — it just shows up on self. When you call r.area(), Rust automatically borrows r as &self. When you call r.scale(2.0), Rust borrows it as &mut self (and requires r to be declared mut).

Debug printing with #[derive]

One immediately practical feature: the #[derive(Debug)] attribute generates a println!-compatible representation for free.

Rust
#[derive(Debug)]
struct Rectangle { width: f64, height: f64 }

fn main() {
    let r = Rectangle { width: 10.0, height: 5.0 };
    println!("{:?}",  r);   // Rectangle { width: 10.0, height: 5.0 }
    println!("{:#?}", r);   // pretty-printed, multi-line
    dbg!(&r);               // prints file + line + value to stderr — great for debugging
}

Habit: add #[derive(Debug)] to every struct you create while learning. You'll want it for dbg!() constantly.

Part 2 — Enums

This is where Rust diverges most sharply from Go — and most significantly from anything in TypeScript. Go's enums are just typed integers (iota). Rust enums are algebraic data types: each variant can carry different data of different types.

Basic enum — like Go's iota

Rust
enum Direction { North, South, East, West }

let d = Direction::North;  // :: to access variants, same as associated functions

Enums with data — unlike anything in Go

Each variant can hold its own distinct payload. This is the feature that makes Rust enums genuinely powerful.

unit variantQuit
tuple variantMove(i32, i32)
named fieldsWrite { text: String }
wrapping a typeChangeColor(u8, u8, u8)
Rust — enum with varied payloads
enum Message {
    Quit,                        // no data
    Move { x: i32, y: i32 },    // named fields, like an inline struct
    Write(String),               // a single String
    ChangeColor(u8, u8, u8),     // three u8 values
}

// All four are the same type: Message.
// You can store them in the same Vec, pass them to the same function.
let msgs: Vec = vec![
    Message::Move { x: 10, y: 20 },
    Message::Write(String::from("hello")),
    Message::Quit,
];

TypeScript comparison: this is a discriminated union — type Message = { kind: 'quit' } | { kind: 'move', x: number, y: number } | ... — but enforced at the type level with no runtime kind field needed, and with exhaustive pattern matching built in.

Pattern matching with match

match is how you work with enum values. It's exhaustive — the compiler requires you to handle every variant. This is the feature that makes "invalid states unrepresentable" a real property of your programs.

Rust — match on an enum with data
fn process(msg: Message) {
    match msg {
        Message::Quit => println!("quit"),
        Message::Move { x, y } => println!("move to ({x}, {y})"),
        Message::Write(text) => println!("write: {text}"),
        Message::ChangeColor(r, g, b) => println!("color: {r},{g},{b}"),
    }
    // If you forget a variant, the compiler errors: "non-exhaustive patterns"
}

match also works on integers, strings, tuples, ranges — it's a general-purpose control flow expression, not just for enums. And unlike a switch statement, every arm is an expression that can return a value:

Rust — match as an expression
let description = match direction {
    Direction::North => "heading north",
    Direction::South => "heading south",
    Direction::East | Direction::West => "heading sideways",  // | for multiple patterns
};

Option<T> — Rust's answer to null

Rust has no null. No nil. No undefined. Instead, the possibility of absence is encoded in the type system using a built-in enum:

Rust — Option definition (from std)
enum Option {
    Some(T),   // there is a value: Some(42), Some("hello"), Some(vec![1,2,3])
    None,      // there is no value
}

If a function might not return a value, it returns Option<T>. If a field might be absent, it's Option<FieldType>. You cannot accidentally use an absent value — the compiler forces you to handle both arms.

Go — nil is invisible in the type
func findUser(id int) *User {
    // ... could return nil
    // caller has no idea from the type alone — must read docs or get a panic
}

user := findUser(42)
user.Name  // potential nil panic — not visible in the type
Rust — absence is in the type
fn find_user(id: u32) -> Option {
    // ... returns Some(user) or None
    // the caller KNOWS from the type that this might be absent
}

match find_user(42) {
    Some(user) => println!("{}", user.name),  // user is a User here — safe
    None       => println!("not found"),
}

// You cannot access user.name without going through the match.
// The compiler won't let you use an Option as if it were a User.

The shift: in Go and TypeScript, null/nil/undefined are values that any pointer or reference type can silently hold. In Rust, absence is a different type — Option<User> is not a User. The type system makes null-safety automatic rather than opt-in.

Common Option methods — avoiding verbose match every time

Rust
let maybe: Option = Some(5);

// unwrap_or: get the value or a default
let x = maybe.unwrap_or(0);        // 5  (or 0 if None)

// map: transform the inner value if Some, leave None alone
let doubled = maybe.map(|v| v * 2); // Some(10)

// is_some / is_none: boolean checks
if maybe.is_some() { println!("has a value"); }

// unwrap: get the value, panic if None — use only when you're certain
let x = maybe.unwrap();             // 5, or panics if None

Avoid .unwrap() in production code. It panics on None — the runtime equivalent of a nil panic. Use .unwrap_or(), .map(), or pattern matching. Lesson 8 covers the full error handling story with Result.

if let — concise single-variant matching

When you only care about one variant and want to ignore the rest, if let is less ceremony than a full match:

Rust
let maybe = Some(42);

// verbose match:
match maybe {
    Some(v) => println!("{v}"),
    None    => {}           // must handle this arm even though you don't care
}

// concise if let:
if let Some(v) = maybe {
    println!("{v}");        // only runs if it's Some
}                           // None is silently ignored — intentional here

Putting it together — a small model

Rust — struct + enum + match working together
#[derive(Debug)]
struct User {
    name:  String,
    email: Option,   // email might not be set
}

impl User {
    fn new(name: &str) -> User {
        User { name: name.to_string(), email: None }
    }

    fn with_email(mut self, email: &str) -> User {   // builder pattern
        self.email = Some(email.to_string());
        self
    }

    fn contact_info(&self) -> String {
        match &self.email {
            Some(e) => format!("{} <{}>", self.name, e),
            None    => self.name.clone(),
        }
    }
}

fn main() {
    let u = User::new("Alice").with_email("alice@example.com");
    println!("{}", u.contact_info());   // Alice 

    let u2 = User::new("Bob");
    println!("{}", u2.contact_info());  // Bob
}

Quiz — no looking back

7 questions

1. In Rust, where do struct methods live?

Correct. Methods live in impl blocks — separate from the struct definition. A struct can have multiple impl blocks (useful for organising large types, and required when implementing traits). This separation is a deliberate design choice: data and behaviour are decoupled at the syntax level.
Rust separates data (struct fields) from behaviour (impl blocks). Methods go in impl StructName { }. This is different from Go, where methods are defined anywhere with a receiver, and from TypeScript classes, where methods are inline. A type can have multiple impl blocks.

2. A method takes &mut self. What does the caller need to do?

Correct. To call a &mut self method, the variable must be declared mut — e.g. let mut r = Rectangle::new(...). Rust then auto-borrows it as &mut self for the call. If the variable isn't mut, the compiler rejects the call: "cannot borrow as mutable."
The caller must declare the variable with let mut. Rust will automatically borrow it as &mut self for the call, but only if mutability was declared. This flows directly from the ownership system: you can only have a mutable borrow if the binding itself is mutable.

3. What makes Rust enums fundamentally different from Go's iota enums?

Correct. Go's iota enums are just typed integers — the variants are constants with no data attached. Rust enum variants are algebraic: Quit carries nothing, Move { x, y } carries two integers, Write(String) carries a String. They're all the same type, but each variant has its own shape. This is what TypeScript calls a discriminated union.
Rust enum variants can each carry different data — zero, named fields, tuple fields, or wrapped types. Go's iota enum variants are bare integer constants with no payload. This single difference makes Rust enums dramatically more expressive for modelling domain state.

4. What happens if you match an enum but forget to handle one variant?

Correct. match is exhaustive by requirement — the compiler errors with "non-exhaustive patterns" if any variant is missing. This is what makes "invalid states unrepresentable" a real guarantee: you can't add a new variant to an enum without updating every match that covers it, ensuring all code paths handle all states.
Compile error — not a runtime panic, not a warning. The compiler checks exhaustiveness statically. This is one of Rust's most valuable features: adding a variant to an enum causes every match on that enum to fail compilation until updated. Unhandled states are impossible to ship.

5. Why can't you access a field on an Option<User> directly, the way you'd access a field on a *User in Go?

Correct. Option<User> and User are distinct types. The compiler will not let you use an Option as if it were its inner type — you must pattern match (or use methods like .map() / .unwrap_or()) to get a User out. This is what makes null safety automatic: there's no way to accidentally skip the None check.
Option<User> is a distinct type from User. You can't call user.name on an Option<User> any more than you can call methods on an i32 that you wanted to be a String. The type system enforces the None check — you must match or use Option methods to extract the inner value.

6. When should you use if let instead of match?

Correct. if let is syntactic sugar for a match with one interesting arm and a catch-all that does nothing. Use match when you care about multiple (or all) variants. Use if let when you only want to act on one and intentionally ignore the others. Works on any enum, not just Option.
if let is concise syntax for "match this one pattern, ignore everything else." Use it when only one variant matters to you and you're deliberately ignoring the rest. It works on any enum (not just Option) and there's no performance difference from match.

7. Why should you avoid .unwrap() on Option in production code?

Correct. .unwrap() on None calls panic!() — the program crashes with an unrecoverable error. It's useful in tests, prototypes, and "this provably can't be None" situations where you'd rather panic than add boilerplate. In production code, prefer .unwrap_or(), .map(), or the ? operator (Lesson 8) to handle None gracefully.
.unwrap() panics on None — same outcome as a nil pointer dereference in Go, just with a clearer error message. It moves the value out (so the Option is consumed), but the real concern is the panic. Prefer .unwrap_or(), .map(), or ? in production paths.

What's next

Structs and enums are Rust's data modelling tools. The next question is behaviour: how do you express that a type can do something — and how do you write code that works across multiple types? That's traits. They'll feel immediately familiar if you know Go interfaces, with one meaningful twist: traits are implemented explicitly, and the compiler checks them at compile time with far more precision.
